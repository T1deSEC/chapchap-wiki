루틴 충돌 분석 에이전트

LangGraph StateGraph로 성분 충돌을 감지하고 사용자 친화적 피드백을 생성하는 원리

에이전트 개요

루틴 충돌 분석은 LangGraph StateGraph로 구현됩니다. POST /analyze/routine 엔드포인트가 요청을 받으면 그래프를 새로 컴파일·실행합니다. 상태는 단일 실행에만 존재하고 세션 간 공유되지 않습니다.

항목 내용
그래프 유형StateGraph(RoutineAnalysisState)
LLM 모델gpt-4o-mini (환경변수 LLM_MODEL으로 오버라이드 가능)
도구get_product_ingredients, find_potentially_conflicting_ingredients
에이전트 패턴첫 노드만 ReAct (create_react_agent), 이후 노드는 단순 LLM 호출

StateGraph 구조

입력: productIds[] + routinePeriod + userSkinType

  [collect_conflicts]  ←── 진입점 (ReAct 에이전트)
         │
         ▼ _route_after_conflicts
    ┌────┴────────────┬──────────────────┐
    │ has_conflicts   │ has_caution      │ no_conflicts
    ▼                 ▼                  ▼
[generate_      [recommend_       [recommend_
 recommendation] complementary]   complementary]
    │
    ▼
[generate_adjustments]
    │
    ▼
   END

노드별 역할

① collect_conflicts (ReAct 에이전트)

그래프의 유일한 도구 활용 노드입니다. 사전에 빌드된 ReAct 에이전트(create_react_agent)를 재사용하고, 결과 JSON을 파싱해 상태를 결정합니다.

  1. get_product_ingredients로 각 제품의 전 성분 목록 조회 (concentration_order 포함)
  2. find_potentially_conflicting_ingredients로 후보 성분 상세 정보 확인
  3. LLM이 충돌 여부 최종 판단 → JSON 반환
  4. 코드가 JSON을 파싱해 conflicting_pairs와 has_caution 추출
ReAct 에이전트 인스턴스(_agent)는 모듈 전역에서 하나만 생성해 재사용합니다. 상태가 없으므로 동시 요청 간 공유해도 안전합니다.

② generate_recommendation (conflict 경로)

발견된 충돌 쌍을 하나씩 구체적으로 짚어가며 권장 사용 방법을 생성합니다. LLM 응답은 raw 텍스트(JSON 아님)로 받아 그대로 recommendation 필드에 저장합니다.

  • 충돌 쌍 수에 비례해 max_tokens를 동적으로 늘립니다 (2048 + len(pairs) × 200)
  • 한국어명 + INCI 영문명 형식("히알루론산(HYALURONIC ACID)")은 DB에서 일괄 조회해 LLM 응답 파싱 후 코드가 붙입니다

③ generate_adjustments (conflict 경로)

앞 노드의 recommendation과 내용이 어긋나지 않도록, 실제 제품명을 활용한 구체적 루틴 조정 방안을 JSON 배열로 생성합니다.

["비타민C 세럼은 아침에, 레티놀 크림은 저녁에 나누어 사용해보세요.",
 "AHA 토너 사용 직후 레티놀 제품 적용은 피해주세요."]

④ recommend_complementary (caution / good 경로)

충돌이 없거나 주의 수준일 때 진입하는 노드입니다. 현재 루틴에 부족한 효과를 파악해 보완 성분을 제안합니다.

진입 조건 첫 문장 맥락
has_caution = true직접 충돌은 없지만 주의 필요 성분 조합이 있음을 부드럽게 전달
has_caution = false같이 쓰기 까다로운 조합이 없어 안심하고 사용해도 된다고 안내

상태(status) 결정 로직

상태는 LLM이 직접 결정하지 않습니다. collect_conflicts가 반환한 두 값으로 코드가 결정론적으로 파생합니다.

// _route_after_conflicts 함수
conflicting_pairs가 비어있지 않으면 → "conflict"
비어있고 has_caution = true → "caution"
비어있고 has_caution = false → "good"
LLM이 JSON에서 "conflicting_pairs"에 실제 쌍을 넣으면 자동으로 conflict가 됩니다. 빈 배열이면 has_caution 플래그가 결정권을 가집니다. LLM의 자연어 "status" 판단에 의존하지 않아 일관성을 확보합니다.

알림 트리거

분석 결과는 Spring Boot AiAnalysisService가 Python 응답을 받은 뒤 처리합니다.

status 알림 타입 DB 저장
conflictROUTINE_CONFLICTnotifications 테이블에 INSERT
cautionROUTINE_CAUTIONnotifications 테이블에 INSERT
good없음알림 없음

상태 필드 전체 목록

필드 타입 초기값 / 설명
routine_periodstr요청의 routinePeriod (AM/PM)
user_skin_typestr요청의 userSkinType
products_strstrDB 조회한 "제품 ID: 제품명" 리스트 문자열
statusstr초기값 "good" → collect_conflicts에서 결정
has_cautionbool충돌은 없지만 피부 타입 기준 주의 성분 조합 존재 여부
conflicting_pairslist[dict]{ingredient1, ingredient2, reason} 형식
recommendationstr사용자에게 보여줄 권장 메시지
suggested_adjustmentslist[str]실행 가능한 루틴 조정 방안 목록

설계 원칙

  • 결정론적 상태 파생 — status는 LLM 자연어 판단이 아니라 코드가 conflicting_pairs 유무로 결정합니다. LLM이 프롬프트에 따라 의도치 않은 "conflict"를 반환해도 실제 쌍이 없으면 무시됩니다.
  • 한국어명 후처리 — LLM에게 INCI 영문명만 반환하게 하고, 한국어 매핑은 DB에서 일괄 조회합니다. LLM이 잘못된 한국어명을 생성하는 hallucination을 방지합니다.
  • ReAct 에이전트 범위 제한 — 도구 호출이 필요한 collect_conflicts만 ReAct 패턴을 사용합니다. 이후 노드는 단순 LLM 호출로 비용을 절감하고 예측 가능성을 높입니다.
