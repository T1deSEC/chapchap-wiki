성분 랭킹 알고리즘

개인화 제품 추천에 사용되는 match_score 계산 원리

핵심 전제: concentration_order

화장품 전성분 표기는 식품의약품안전처 규정상 함량이 높은 성분부터 먼저 기재합니다. 따라서 concentration_order 값이 작을수록 제품 내 해당 성분의 함량이 높습니다.

concentration_order = 1 → 함량 1위 (가장 많음)
concentration_order = 20 → 함량 20위 (소량)

이 성질을 이용하면 "추천 성분이 얼마나 많이 들어있는가"를 숫자로 표현할 수 있습니다.

match_score 공식

match_score = (count_matching × 2.0) + Σ( 1 / concentrationOrderi )

두 항의 의미:

  • count_matching × 2.0 — 추천 성분이 제품에 몇 개 포함되어 있는지 (성분 수 기여)
  • Σ(1 / concentrationOrder) — 각 추천 성분의 함량 순위 역수 합 (고함량 성분에 더 높은 가중치)

계산 예시

시나리오 count × 2.0 Σ(1/order) match_score
추천 성분 3개 (5·10·20위) 3 × 2.0 = 6.0 0.20 + 0.10 + 0.05 = 0.35 6.35
추천 성분 1개 (1위 고함량) 1 × 2.0 = 2.0 1 / 1 = 1.00 3.00
추천 성분 2개 (3·8위) 2 × 2.0 = 4.0 0.33 + 0.13 = 0.46 4.46
시나리오 2 (성분 1개, 1위)보다 시나리오 3 (성분 2개, 3·8위)이 더 높은 점수를 받습니다. 추천 성분을 많이 포함하면서 고함량일수록 상위에 랭크됩니다.

기피 성분 처리

기피 성분(ingredientsToAvoid)이 하나라도 포함된 제품은 match_score와 무관하게 완전 제외됩니다. 알레르기나 피부 악화 가능성이 명확한 경우에만 기피 성분으로 등록하도록 LLM 에이전트에 지시합니다.

임계 점수(threshold)를 두지 않습니다. 과도한 필터링으로 추천 결과가 0개가 되는 사태를 방지하기 위한 설계 결정입니다.

AI 에이전트 성분 선정 원칙

  1. 핵심 성분 집중 — 추천 성분은 3~5개로 제한. 너무 많으면 match_score 변별력이 떨어집니다.
  2. DB 실존 성분만 추천 — search_ingredients_by_function 툴로 DB에 실제 존재하는 성분만 선정. LLM hallucination으로 존재하지 않는 INCI명을 추천하면 match_score 계산이 불가능합니다.
  3. 기피 성분 최소화 — 알레르기나 피부 악화 가능성이 명확히 문서화된 케이스만 등록. 일반적인 "좋지 않은" 성분을 전부 기피로 넣으면 추천 결과가 너무 줄어듭니다.

성분 추천 에이전트 전체 흐름

match_score는 추천 에이전트의 마지막 단계에서 제품 순위를 결정할 때 사용됩니다.

1. LLM이 피부 타입·고민으로 추천 성분 INCI명 후보 선정 (5~8개)
2. DB 검증 — DB에 실존하는 성분만 통과 (get_ingredients_by_inci_names, 코드 직접 호출)
3. LLM이 기피 성분 후보 선별 → 최종 추천·기피 성분 확정
4. Spring Boot: DB에 결과 저장 → INGREDIENT_RECOMMENDATION 알림 발송
5. 룰 기반: 각 제품에서 match_score 계산 → 상위 제품 추천 목록 반환
성분 추천 에이전트는 LangGraph StateGraph가 아닌 순차 함수 파이프라인으로 구현됩니다. 단방향 흐름이고 분기가 없어 Graph 구조가 필요하지 않습니다.

match_score 계산 코드는 Spring Boot 서버(IngredientRecommendationService)에 있습니다. Python AI 서비스가 추천 성분 목록을 반환하면, Spring Boot가 DB의 제품들을 순회하며 각 제품에 대해 match_score를 산출하고 순위가 높은 제품만 응답합니다.
