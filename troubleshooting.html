로컬 개발 및 배포 환경에서 자주 발생하는 문제와 해결 방법

Docker 메모리 부족 (OOM) — t2.micro

AWS EC2 t2.micro(1GB RAM)에서 docker-compose --profile full up을 실행하면 Spring Boot JVM이 메모리를 과점하거나 OOM Killer가 컨테이너를 강제 종료합니다.

증상: 컨테이너가 갑자기 죽거나, docker logs chapchap-backend에서 Killed 메시지, 또는 Spring Boot가 시작 중 멈춤.

해결 방법 — docker-compose.yml의 Spring Boot 서비스에 JVM 힙 제한을 추가합니다.

services:
  backend:
    environment:
      - JAVA_OPTS=-Xmx256m -Xms128m
Python AI 서비스도 일부 메모리를 사용합니다. 분석 요청 중 모델 로딩으로 일시적으로 spike가 발생할 수 있습니다. 전체 스택을 안정적으로 운영하려면 t2.small(2GB) 이상을 권장합니다.

docker-compose --profile full 빌드 실패

Docker 이미지 빌드 시 COPY build/libs/*.jar app.jar가 실패합니다.

증상: failed to solve: failed to read dockerfile 또는 COPY failed: file not found in build context

원인 — Gradle이 JAR를 아직 생성하지 않은 상태에서 Docker 빌드를 시도했습니다.

해결 방법 — Docker Compose 실행 전에 반드시 bootJar를 먼저 빌드합니다.

.\gradlew.bat bootJar
docker-compose --profile full up --build -d

Flyway 체크섬 오류로 Spring Boot 기동 실패

이미 적용된 마이그레이션 파일(V1~V4)을 수정하면 다음 기동 시 Flyway가 체크섬 불일치를 감지해 예외를 던집니다.

FlywayException: Validate failed: Migrations have failed validation
Migration checksum mismatch for migration version 3

해결 방법 (개발 환경)

# application.properties 또는 application-local.yml
spring.flyway.validate-on-migrate=false
# 또는 한 번만 repair 실행 후 원복
spring.flyway.repair-on-migrate=true
가장 근본적인 해결책은 기존 V1~V4 파일을 절대 수정하지 않는 것입니다. 스키마 변경이 필요하면 반드시 V5, V6… 새 파일로 추가합니다.

회원가입 시 403 Forbidden

POST /api/auth/register가 403을 반환합니다.

{ "success": false, "data": null, "message": "유효하지 않은 가입 코드입니다" }

원인 — 요청의 registrationToken이 서버의 REGISTRATION_SECRET 환경변수 값과 일치하지 않습니다.

확인 사항

  • .env 파일에 REGISTRATION_SECRET=<가입 코드>가 정의되어 있는지 확인
  • Docker Compose 환경에서는 env_file: .env가 설정되어 있는지 확인
  • REGISTRATION_SECRET을 빈 문자열로 설정하면 누구든 가입 가능해지므로 주의
# .env 예시
REGISTRATION_SECRET=my-secret-signup-code
환경변수 이름은 REGISTRATION_TOKEN이 아니라 REGISTRATION_SECRET입니다. Spring의 @Value("${app.registration.secret:}")가 이 값을 바인딩합니다.

Spring Boot 기동 실패 — JWT 시크릿 너무 짧음

Spring Boot가 기동하자마자 예외를 던지고 종료됩니다.

WeakKeyException: The signing key's size is 120 bits which is not secure enough...
jjwt는 HMAC-SHA256에 최소 256비트(32문자) 이상의 시크릿을 요구합니다.

해결 방법 — .env의 JWT_SECRET을 32자 이상으로 설정합니다.

# .env 예시 (32자 이상)
JWT_SECRET=my-super-secret-key-at-least-32chars!!

AI 서비스 422 Unprocessable Entity

Spring Boot에서 Python AI 서비스(:8000)로 보낸 요청이 422로 거부됩니다.

Spring Boot 로그: AI 서비스 호출 실패: 422
Python 로그: 422 Unprocessable Entity: value is not a valid dict

원인 — Pydantic 모델에 camelCase↔snake_case 변환 설정이 누락되어 있습니다. Spring Boot는 camelCase JSON을 보내는데, Pydantic이 snake_case만 받기 때문입니다.

해결 방법 — Pydantic 모델과 FastAPI 엔드포인트 모두에 설정이 필요합니다.

# schemas/models.py
from pydantic import BaseModel, ConfigDict
from pydantic.alias_generators import to_camel

class MyModel(BaseModel):
    model_config = ConfigDict(
        alias_generator=to_camel,
        populate_by_name=True  # snake_case 직접 접근도 허용
    )
    my_field: str
# main.py (FastAPI 엔드포인트)
@app.post("/analyze/ingredient")
async def analyze_ingredient(req: IngredientAnalysisInput) -> IngredientAnalysisOutput:
    result = analyze_ingredients(req)
    return result  # response_model_by_alias=True 필요

# 또는 엔드포인트 레벨에서 명시
@app.post("/analyze/ingredient", response_model_by_alias=True)
async def analyze_ingredient(req: IngredientAnalysisInput):
    ...

AI 분석 요청 타임아웃 (502 / 504)

AI 분석 API 호출이 일정 시간 후 502 Bad Gateway 또는 504 Gateway Timeout을 반환합니다.

AI 성분 분석(POST /api/analysis/ingredient)은 성분 수에 따라 30~90초가 소요될 수 있습니다. nginx 기본 proxy_read_timeout은 60초입니다.

해결 방법 — nginx 설정에서 AI 경로에 대해 타임아웃을 늘립니다.

location /api/analysis/ {
    proxy_pass http://backend:8080;
    proxy_read_timeout 120s;
    proxy_connect_timeout 10s;
    proxy_send_timeout 120s;
}
현재 프로젝트의 nginx 설정에는 이미 proxy_read_timeout 120s가 적용되어 있습니다. 성분이 매우 많은 제품에서 여전히 타임아웃이 발생한다면 값을 더 늘리거나, Spring Boot WebClient의 타임아웃도 함께 조정하세요.

AI 분석 API 503 Service Unavailable

POST /api/analysis/ingredient 또는 /api/analysis/routine가 503을 반환합니다.

{ "success": false, "message": "잘못된 입력입니다" }
실제 원인은 Python AI 서비스(:8000)가 응답하지 않는 것입니다.

확인 사항

  • docker-compose --profile full ps에서 Python 컨테이너 상태 확인
  • curl http://localhost:8000/health로 AI 서비스 직접 헬스체크
  • OPENAI_API_KEY가 설정되어 있는지 확인 (미설정 시 서비스 기동 후 첫 요청에서 실패)

브라우저 CORS 오류 (배포 환경)

로컬 개발에서는 Vite 프록시가 CORS를 우회하므로 이 문제가 발생하지 않습니다. 프로덕션에서 프론트엔드 도메인이 다를 때 발생합니다.

Access to XMLHttpRequest at 'https://api.example.com' from origin 'https://app.example.com' has been blocked by CORS policy

해결 방법 — .env에 허용 오리진을 설정합니다.

# .env 예시 (쉼표로 구분)
CORS_ALLOWED_ORIGINS=https://app.example.com,https://www.example.com
Spring Boot WebConfig가 CORS_ALLOWED_ORIGINS 환경변수를 읽어 allowedOrigins로 설정합니다. 값이 없으면 CORS 허용 오리진이 비어있어 모든 브라우저 요청이 차단됩니다.
