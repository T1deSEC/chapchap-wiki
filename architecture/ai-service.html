AI 서비스

FastAPI + LangGraph 기반 AI 마이크로서비스의 전체 설계 — 에이전트 그래프, 상태, 라우팅, 알고리즘, Spring Boot 연동까지.

서비스 개요

CHAPCHAP AI 서비스는 Spring Boot 백엔드와 분리된 Python 마이크로서비스입니다. FastAPI(:8000)를 HTTP 인터페이스로 사용하며, 내부에서 LangGraph StateGraph와 OpenAI gpt-4o-mini를 조합해 세 가지 AI 기능을 제공합니다.

기능 구현 방식 엔드포인트
성분 분석 LangGraph StateGraph (4-node, 병렬 Send) POST /analyze/ingredient
루틴 충돌 진단 LangGraph StateGraph (4-node, ReAct agent) POST /analyze/routine
성분 추천 순차 파이프라인 (StateGraph 없음) POST /recommend/ingredients
헬스체크 — GET /health

모든 POST 엔드포인트는 Pydantic 모델의 alias_generator=to_camel과 response_model_by_alias=True를 통해 Spring Boot와 camelCase JSON으로 통신합니다. 오류 발생 시 product_id·skin_type 컨텍스트를 로깅한 후 HTTPException(500)을 반환합니다.

Pydantic 스키마

모든 모델은 다음 설정을 공유합니다.

_camel = ConfigDict(alias_generator=to_camel, populate_by_name=True)

성분 분석 — StateGraph 상세

ingredient_agent.py가 LangGraph StateGraph를 사용해 성분 분석을 수행합니다. 노드 4개로 구성되며, 두 번째 노드는 Send API를 통해 성분별로 병렬 실행됩니다.

State 필드

class IngredientAnalysisState(TypedDict):
    product_id: int
    user_skin_type: str
    user_skin_concerns: list[str]
    triaged: list[dict]                                    # 분석 대상 선별된 성분 목록
    all_ingredients: list[dict]                            # 제품의 전체 성분 목록
    ingredient_analyses: Annotated[list[dict], operator.add]  # 병렬 결과 누적
    final_ingredient_analyses: list[dict]                  # 점수 계산 완료 후 최종 목록
    safety_score: int
    summary: str
    recommendations: list[str]
    skin_impacts: list[dict]
Annotated[list[dict], operator.add]는 LangGraph의 병렬 노드 결과 병합 패턴입니다. Send API로 동시에 실행된 analyze_one_ingredient 노드들의 반환값이 이 필드에 자동으로 누적됩니다.

그래프 플로우

START
  └─► collect_and_triage
        └─► [Send × N] analyze_one_ingredient  ← 성분별 병렬 실행
              └─► aggregate_score
                    └─► generate_summary
                          └─► END

노드 1: collect_and_triage

DB에서 제품의 전체 성분 목록을 concentration_order 순으로 조회합니다. 이후 LLM에게 사용자 피부 타입·고민과 대조하여 분석에 의미 있는 성분 3~10개를 선별하도록 요청합니다.

LLM이 유효하지 않은 성분명을 반환하거나 실패할 경우 폴백으로 상위 5개(concentration_order 기준)를 자동 선택합니다.

노드 2: analyze_one_ingredient (병렬)

라우터 함수 _route_to_ingredient_analyses가 triaged 목록을 순회하며 각 성분에 대해 Send("analyze_one_ingredient", {...})를 반환합니다. LangGraph가 이를 동시에 실행합니다.

각 호출에서 LLM은 해당 성분 하나에 대해 다음을 반환합니다.

  • concern_score (0–100): 피부 우려 점수. 높을수록 해당 성분이 이 사용자에게 부적합.
  • assessment: "적합" | "주의" | "부적합"
  • reason: 평가 근거 (한국어)

LLM 설정: gpt-4o-mini, max_retries=2, timeout=60.

노드 3: aggregate_score

병렬 분석 결과를 모아 제품 전체 안전도 점수를 계산합니다. 계산 로직은 안전도 점수 계산 페이지에 상세히 기술되어 있습니다. 핵심 공식은 다음과 같습니다.

weight = 1 / concentration_order
multiplier: 적합=0.5, 주의=1.0, 부적합=1.5
ingredient_score = clamp(100 − concern_score × multiplier, 0, 100)
safety_score = Σ(ingredient_score × weight) / Σ(weight)

각 성분의 safety_level은 임계값 두 개로 결정됩니다.

  • CONCERN_SAFE_THRESHOLD = 34 이하 → "적합"
  • CONCERN_WARNING_THRESHOLD = 67 이하 → "주의"
  • 그 초과 → "부적합"

노드 4: generate_summary

LLM이 성분 분석 결과 전체를 바탕으로 종합 요약(summary)과 사용 권고사항(recommendations)을 생성합니다. 이와 함께 11개 피부 영향 카테고리에 대한 skin_impacts 점수를 계산합니다.

skin_impacts 이중 점수 체계

각 카테고리 점수는 두 항목의 합으로 구성됩니다.

ingredient_contribution (0–60)
+ 선별 성분들이 해당 카테고리에 실제로 기여하는 정도

user_relevance (0–40)
+ 해당 카테고리가 이 사용자의 피부 타입·고민에 얼마나 관련 있는가

category_score = ingredient_contribution + user_relevance

score가 0인 카테고리는 최종 skin_impacts 배열에서 제외됩니다. score 60 이상이면 "높음"(녹색), 30 이상이면 "중간"(주황), 미만이면 "낮음"(빨강)으로 분류합니다.

카테고리 ≥ 60 30–59 < 30
보습높음중간낮음
피부 장벽높음중간낮음
미백·톤업높음중간낮음
항산화높음중간낮음
진정·항염높음중간낮음
각질케어높음중간낮음
항노화·재생높음중간낮음
탄력높음중간낮음
피지·모공높음중간낮음
항균·트러블케어높음중간낮음
자외선 차단높음중간낮음

skin_impacts는 Spring Boot의 user_product_ai_analysis 테이블 skin_impacts 컬럼(JSONB)에 저장됩니다. JPA 엔티티에서 @JdbcTypeCode(SqlTypes.JSON) + String 타입으로 선언하며, Jackson이 직렬화·역직렬화를 담당합니다.

루틴 분석 — StateGraph 상세

routine_agent.py가 LangGraph StateGraph로 루틴 충돌 진단을 수행합니다. 성분 분석과 달리 첫 번째 노드에 ReAct 에이전트를 사용해 도구를 능동적으로 호출합니다.

State 필드

class RoutineAnalysisState(TypedDict):
    routine_period: str           # "AM" | "PM"
    user_skin_type: str
    products_str: str             # 제품명 목록 (LLM 컨텍스트용)
    status: str                   # "good" | "caution" | "conflict"
    has_caution: bool
    conflicting_pairs: list[dict]
    recommendation: str
    suggested_adjustments: list[str]

ReAct 에이전트 (싱글턴)

collect_conflicts 노드에서 사용하는 ReAct 에이전트는 모듈 임포트 시 한 번만 생성되어 재사용됩니다.

_agent = create_react_agent(_llm(), TOOLS)  # 싱글턴 — 모듈 로드 시 1회 생성

에이전트에 등록된 도구는 두 가지입니다.

  • get_product_ingredients(product_id): 제품의 전 성분을 concentration_order 순으로 반환. ingredient_id도 포함해 다음 도구로 연결 가능.
  • find_potentially_conflicting_ingredients(ingredient_id_list): 성분 ID 목록의 상세 정보(INCI명, 한글명, 설명)를 반환. LLM이 이 정보로 충돌 여부를 직접 판단.

그래프 플로우 및 조건부 라우팅

START
  └─► collect_conflicts (ReAct)
        ├─ conflicting_pairs 있음 ──► generate_recommendation
        │                                 └─► generate_adjustments ──► END
        └─ 없음 (good/caution) ────► recommend_complementary ──► END

라우터 함수 has_conflicts가 state["conflicting_pairs"]의 존재 여부를 확인해 분기를 결정합니다.

노드 1: collect_conflicts

ReAct 에이전트가 루틴 내 제품들을 순회하며 도구를 자율적으로 호출, 성분 충돌 후보를 탐색합니다. 에이전트가 JSON 형식으로 결과를 반환합니다.

{ "conflicting_pairs": [...], "has_caution": true }

반환 직후 한국어 이름 후처리가 실행됩니다. LLM은 INCI명만 반환하므로, 코드가 DB에 배치 쿼리(_ko_name_map())를 보내 모든 충돌 쌍의 성분에 한글명을 추가합니다.

상태값은 코드가 결정론적으로 계산합니다. LLM이 판단하지 않습니다.

status = "conflict" if pairs else ("caution" if has_caution else "good")

노드 2: generate_recommendation (충돌 경로)

충돌 쌍별로 자연어 설명을 생성합니다. 왜 해당 성분 조합이 문제인지, 어떤 반응이 우려되는지 LLM이 설명합니다. max_tokens는 충돌 쌍 수에 비례해 스케일됩니다.

max_tokens = 2048 + len(conflicting_pairs) × 200

노드 3: generate_adjustments (충돌 경로)

제품명 기반으로 구체적인 조정 방안을 제안합니다. 충돌 쌍이 많더라도 사용자 피로를 줄이기 위해 최대 3개로 제한합니다.

n_items = min(len(conflicting_pairs), 3)

노드 (대안 경로): recommend_complementary

충돌이 없는 경우(good/caution)에 실행됩니다. 현재 루틴을 보완할 수 있는 성분이나 제품 유형을 제안합니다.

성분 추천 — 순차 파이프라인

성분 추천은 StateGraph를 사용하지 않습니다. 세 단계의 함수 호출이 순서대로 실행되는 단순한 파이프라인입니다.

def recommend_ingredients(req: IngredientRecommendationInput):
    # 1단계: LLM이 INCI명 5~8개 선택
    inci_names = _select_ingredients_by_llm(req)

    # 2단계: DB에 실제로 존재하는 성분만 필터링
    verified = _verify_ingredients_in_db(inci_names)

    # 3단계: 피해야 할 성분 후보 (DB 상위 100개) + LLM 최종 포매팅
    avoid_candidates = _get_all_ingredients_from_db()
    return _format_recommendation_by_llm(req, verified, avoid_candidates)
  • 1단계: LLM에게 사용자의 피부 타입·고민에 적합한 INCI명 5~8개를 요청합니다.
  • 2단계: DB에 없는 성분명은 모두 제거합니다. LLM이 hallucination으로 존재하지 않는 성분을 반환해도 DB 필터가 방어합니다.
  • 3단계: DB에서 검증된 추천 성분과 피해야 할 성분 후보를 모두 LLM에 전달해 최종 결과물(추천 목록·피해야 할 목록·요약)을 생성합니다.
2단계 DB 검증이 추천 품질의 핵심 안전장치입니다. CHAPCHAP DB에 등록된 성분만 추천에 포함되므로, 사용자는 플랫폼이 실제로 데이터를 보유한 성분만 추천받습니다.

Spring Boot ↔ Python 통신

React 앱
  └─► POST /api/analysis/ingredient  (Spring Boot :8080)
        └─► AiAnalysisService (WebClient, 비동기)
              └─► POST /analyze/ingredient  (Python FastAPI :8000)
                    └─► LangGraph + OpenAI gpt-4o-mini
                          └─► IngredientAnalysisOutput (camelCase JSON)
                                └─► AiAnalysisService 파싱 → DB 저장
                                      └─► ApiResponse
항목 내용
Spring HTTP 클라이언트 WebClient (비동기, Reactor 기반)
직렬화 형식 JSON, camelCase (Pydantic alias_generator=to_camel)
Python 응답 직렬화 FastAPI response_model_by_alias=True
AI 서비스 오류 처리 Python에서 HTTPException(500) → Spring이 BusinessException으로 래핑
AI 서비스 접근 Spring Boot에서만 직접 호출 — 프론트는 Spring API를 통해 간접 접근
Python 모델에 ConfigDict(alias_generator=to_camel, populate_by_name=True)와 엔드포인트에 response_model_by_alias=True가 없으면 Spring이 받는 JSON 키가 snake_case가 되어 역직렬화 실패(필드 누락)로 이어집니다.

성능 설계 포인트

  • 병렬 Send API: 성분 분석에서 성분별 GPT 호출을 순차가 아닌 동시 실행. 성분 10개 기준 직렬 대비 응답 시간이 선형 단축됩니다.
  • ReAct 에이전트 싱글턴: create_react_agent 객체는 모듈 로드 시 1회만 생성합니다. 요청마다 새로 만들면 LLM 클라이언트 초기화 비용이 반복됩니다.
  • 한국어 이름 배치 조회: INCI명 하나씩 DB를 조회하지 않고 _ko_name_map()으로 전체 이름을 한 번에 조회합니다.
  • max_tokens 동적 스케일링: 루틴 충돌 수에 따라 토큰 한도를 조정해 짧은 루틴에서는 불필요한 토큰을 낭비하지 않습니다.
  • LLM max_retries=2, timeout=60: OpenAI API 일시 장애 시 자동 재시도하되, 60초 이상 대기하지 않습니다.
