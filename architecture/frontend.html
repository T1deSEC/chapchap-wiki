상태 관리 (Zustand)

useAuthStore

JWT 토큰과 사용자 정보를 관리하는 전역 스토어입니다. localStorage에 영속되어 새로고침 후에도 로그인 상태가 유지됩니다.

항목 값
파일src/store/authStore.ts
영속localStorage (chapchap-auth)
필드accessToken, refreshToken, user, isAuthenticated
액션login(), setTokens(), logout(), updateUser()

useAnalysisStore

AI 분석 2단계 플로우(로딩 페이지 → 결과 페이지) 브릿지 역할을 하는 스토어입니다. 로딩 페이지가 API 호출 후 결과를 저장하면 결과 페이지가 읽어갑니다. 결과 소비 후 스토어를 클리어해야 합니다. 인메모리 전용이므로 새로고침 시 결과가 소멸됩니다.

항목 값
파일src/store/analysisStore.ts
영속인메모리 (새로고침 시 소멸)
필드routineResult: RoutineAnalysisResult | null

API 레이어

모든 HTTP 요청은 src/api/client.ts의 단일 Axios 인스턴스를 통해 처리됩니다. 도메인별 모듈(auth.ts, home.ts, ingredient.ts, routine.ts, profile.ts, products.ts, notifications.ts)이 실제 API 함수를 정의합니다.

Request 인터셉터

모든 요청에 Authorization: Bearer <accessToken> 헤더를 자동 첨부합니다. useAuthStore에서 토큰을 읽어 주입하므로 각 API 함수에서 별도 처리가 필요 없습니다.

Response 인터셉터: unwrapApiResponse

모든 백엔드 응답의 { success, data, message } 래퍼를 자동으로 벗겨냅니다. 훅에서 r.data는 이미 실제 페이로드이므로 추가로 .data를 한 번 더 접근하면 undefined가 됩니다.

Response 인터셉터: 401 자동 처리

액세스 토큰 만료 시 자동으로 갱신을 시도하고 원본 요청을 재시도합니다. 복수 요청이 동시에 401을 수신해도 갱신 요청은 한 번만 발생합니다.

401 수신
  └─► isRefreshing 플래그 확인
        ├─► 이미 갱신 중: failedQueue에 원본 요청 추가
        └─► 갱신 시작: POST /api/auth/refresh
              ├─► 성공: 새 토큰 저장 → failedQueue 모든 요청 재시도
              └─► 실패: logout() → /login 리다이렉트

TanStack Query 설정

서버 데이터 패칭·캐싱은 TanStack Query v5로 관리합니다. 훅은 src/hooks/에 도메인별로 분리되며, 페이지 컴포넌트는 훅을 통해서만 API 함수를 사용합니다.

설정 값
staleTime5분 (1000 × 60 × 5)
retry1
설정 위치src/main.tsx

주요 타입 대응표

모든 공유 타입은 src/types/index.ts 한 파일에서 관리합니다. 백엔드 DTO와 1:1 대응합니다.

프론트 타입 백엔드 DTO 주요 필드
DiaryEntryDiaryResponselogDate, skinStatus, memo
IngredientItemProductIngredientResponseinciName, koName, functionTags, concentrationOrder
RoutineItemRoutineProductItemstepOrder, productName, brand, imageUrl
RoutineRecordRoutineResponseGET /api/routine?period=AM|PM, 루틴 없으면 404→null
AiIngredientResult 내 성분IngredientAnalysisinciName, koName, assessment, reason
FeedbackRecordFeedbackResponseeffective (isEffective 아님)
WishlistItemWishlistItemResponseproductId (단일 id 없음)
PageResponse<T>Spring Page<T>.content[], .page.totalElements
주의: 필드명은 백엔드 DTO 기준입니다. 예를 들어 일기의 날짜 필드는 logDate이며 date가 아닙니다. FeedbackRecord.effective는 boolean이며 isEffective가 아닙니다.

프로덕션 빌드 & 배포

프로덕션에서는 npm run build로 생성된 정적 파일을 EC2의 nginx가 서빙합니다. 별도 호스팅 서비스 없이 백엔드와 같은 서버에서 운용됩니다.

빌드 & 배포 흐름

npm run build              # dist/ 생성
rm -rf ~/CHAPCHAP-Backend/frontend-dist/*
cp -r dist/* ~/CHAPCHAP-Backend/frontend-dist/
docker compose -f ~/CHAPCHAP-Backend/docker-compose.yml restart nginx

nginx 컨테이너가 frontend-dist/를 read-only 볼륨으로 마운트하므로, 파일 교체 후 nginx만 재시작하면 됩니다. Spring Boot 재빌드는 불필요합니다.

API Base URL 설정

src/api/client.ts의 Axios 인스턴스는 VITE_API_BASE_URL 환경변수를 baseURL로 사용합니다. 로컬과 프로덕션 모두 빈값으로 설정하며, 요청 경로는 /api/* 상대 경로를 사용합니다.

환경 VITE_API_BASE_URL 처리 주체
로컬 개발빈값Vite dev server 프록시가 /api/* → :8080 전달
EC2 프로덕션빈값nginx가 /api/* → Spring Boot :8080 프록시
Vite 프록시(vite.config.ts의 server.proxy)는 개발 서버에서만 동작합니다. 프로덕션 빌드에서는 nginx가 동일한 역할을 담당하므로 VITE_API_BASE_URL을 별도로 설정할 필요가 없습니다.

개발 전용 기능

로그인 페이지(/login)에 [개발용] 백엔드 없이 로그인 버튼이 있습니다. import.meta.env.DEV === true일 때만 렌더링되며, API 호출 없이 useAuthStore에 mock 상태를 주입합니다. 프로덕션 빌드(NODE_ENV=production)에서는 표시되지 않습니다.
