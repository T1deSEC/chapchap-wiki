파이프라인 명세

멀티 서비스 비동기 파이프라인의 생명주기 — 성공·실패 분기 및 예외 처리 흐름. 다이어그램을 클릭하면 확대됩니다.

참여자 범례: 클라이언트(React) → Spring Boot(:8080) → FastAPI AI 서비스(:8000) → PostgreSQL(:5432) → OpenAI API. 각 파이프라인에서 트랜잭션 경계와 서비스 간 예외 전파 경로를 확인할 수 있습니다.

성분 분석 파이프라인

POST /api/analysis/ingredient 요청이 Spring Boot를 거쳐 AI 서비스로 전달되고, LangGraph StateGraph가 4개 노드를 순차·병렬 실행한 뒤 결과가 DB에 저장되는 전체 흐름입니다.

sequenceDiagram autonumber participant C as 클라이언트 participant S as Spring Boot participant AI as FastAPI participant DB as PostgreSQL participant LLM as OpenAI C->>S: POST /api/analysis/ingredient S->>AI: POST /analyze/ingredient (WebClient) AI->>DB: SELECT product_ingredients
WHERE product_id = ? DB-->>AI: 전체 성분 목록 AI->>LLM: collect_and_triage
(관련 성분 3–10개 선별) LLM-->>AI: triaged 성분 목록 Note over AI,LLM: LangGraph Send API — N개 성분 병렬 실행 AI->>LLM: analyze_one_ingredient × N LLM-->>AI: concern_score / assessment / reason (각 성분) Note over AI: aggregate_score (결정론적 가중 평균) Note over AI: safety_score · safety_level 코드 계산 AI->>LLM: generate_summary + skin_impacts LLM-->>AI: summary / recommendations / skin_impacts AI-->>S: IngredientAnalysisOutput (camelCase JSON) S->>DB: INSERT user_product_ai_analysis
(skin_impacts → JSONB) DB-->>S: 저장 완료 S-->>C: ApiResponse<AnalysisResultResponse>

루틴 충돌 진단 파이프라인

POST /api/analysis/routine 요청에서 ReAct 에이전트가 DB 도구를 능동적으로 호출해 충돌을 탐지하고, 결과에 따라 알림이 자동 발송되는 흐름입니다.

sequenceDiagram autonumber participant C as 클라이언트 participant S as Spring Boot participant AI as FastAPI participant DB as PostgreSQL participant LLM as OpenAI participant N as NotificationService C->>S: POST /api/analysis/routine S->>AI: POST /analyze/routine (WebClient) Note over AI,LLM: ReAct 에이전트 (싱글턴) — 도구 자율 호출 loop 제품별 반복 AI->>DB: get_product_ingredients(product_id) DB-->>AI: 성분 목록 (ingredient_id 포함) end AI->>DB: find_potentially_conflicting_ingredients(ids) DB-->>AI: 성분 상세 정보 AI->>LLM: ReAct 추론 — 충돌 판단 LLM-->>AI: conflicting_pairs / has_caution Note over AI: status 결정 (결정론적)
"conflict" | "caution" | "good" AI->>DB: _ko_name_map() 배치 조회 DB-->>AI: 한글 성분명 alt status = "conflict" AI->>LLM: generate_recommendation + generate_adjustments else status = "caution" or "good" AI->>LLM: recommend_complementary end LLM-->>AI: 최종 텍스트 생성 AI-->>S: RoutineAnalysisOutput (camelCase JSON) S->>DB: 분석 결과 저장 DB-->>S: 저장 완료 alt status = "conflict" S->>N: ROUTINE_CONFLICT 알림 생성 else status = "caution" S->>N: ROUTINE_CAUTION 알림 생성 end S-->>C: ApiResponse<RoutineAnalysisResponse>
알림 발송 선행 조건: NotificationService는 DB 저장 성공 이후에만 호출됩니다. AI 호출 성공 후 저장이 실패하면 알림 발송이 방지되어 데이터 불일치를 방지합니다.

성분 추천 파이프라인

POST /recommend/ingredients는 StateGraph 없이 3단계 순차 함수 호출로 구성됩니다. LLM 환각 방지를 위해 DB 검증이 중간 단계에 위치합니다.

sequenceDiagram autonumber participant C as 클라이언트 participant S as Spring Boot participant AI as FastAPI participant DB as PostgreSQL participant LLM as OpenAI C->>S: POST /api/recommendation/ingredients (AI) S->>AI: POST /recommend/ingredients (WebClient) AI->>LLM: _select_ingredients_by_llm
(피부 타입·고민 기반 INCI명 5–8개) LLM-->>AI: INCI 성분명 목록 AI->>DB: _verify_ingredients_in_db
(DB에 없는 성분 필터링) DB-->>AI: 검증된 성분 목록 AI->>DB: _get_all_ingredients_from_db
(기피 성분 후보 top 100) DB-->>AI: 전체 성분 풀 AI->>LLM: _format_recommendation_by_llm
(추천 + 기피 + 요약 생성) LLM-->>AI: IngredientRecommendationOutput AI-->>S: JSON (camelCase) S-->>C: ApiResponse<IngredientRecommendationResponse>
DB 검증 필터 (3단계): LLM이 hallucination으로 존재하지 않는 성분명을 반환해도 _verify_ingredients_in_db가 DB와 대조해 제거합니다. 추천 결과는 항상 플랫폼이 실제로 보유한 성분으로만 구성됩니다.

공통 예외 처리 흐름

세 파이프라인 모두 동일한 예외 전파 구조를 공유합니다. 서비스 경계 어디서 오류가 발생해도 클라이언트는 일관된 ApiResponse 구조를 수신합니다.

sequenceDiagram participant AI as FastAPI participant S as Spring Boot participant GEH as GlobalExceptionHandler participant C as 클라이언트 AI-->>S: HTTPException 500 (분석 실패 · 타임아웃) S->>S: BusinessException 래핑 Note over GEH: @RestControllerAdvice S->>GEH: 예외 전달 alt BusinessException GEH-->>C: HTTP 도메인 정의 코드
ApiResponse.error(message) else MethodArgumentNotValidException GEH-->>C: HTTP 400
ApiResponse.error(검증 실패 메시지) else Exception (그 외) GEH-->>C: HTTP 500
ApiResponse.error("서버 오류") end
모든 오류 응답은 { "success": false, "message": "..." } 구조입니다. 클라이언트는 서비스 내부 구조와 무관하게 동일한 형식으로 오류를 처리할 수 있습니다.
