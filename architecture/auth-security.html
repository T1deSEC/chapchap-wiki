인증 & 보안

JWT 발급·갱신·폐기 시퀀스, 토큰 저장 전략, SecurityConfig 필터 체인, 회원가입 접근 제어.

JWT 전체 시퀀스

[1] 로그인
  사용자 → POST /api/auth/login (email, password)
    └─► Spring Boot: 인증 성공 → JWT 발급
          AccessToken (30분) + RefreshToken (14일) 반환
          RefreshToken은 DB(refresh_tokens 테이블)에도 저장

[2] 인증된 요청
  사용자 → GET /api/diary (+ Authorization: Bearer <AccessToken>)
    └─► JwtAuthenticationFilter: 토큰 검증
          ├─► 유효 → SecurityContext 설정 → 컨트롤러 진행
          └─► 유효하지 않음 → 401 반환

[3] 토큰 갱신
  프론트 axios 인터셉터: 401 수신
    └─► POST /api/auth/refresh (+ RefreshToken)
          ├─► DB 검증: 유효하고 폐기되지 않은 RefreshToken인지 확인
          │     └─► 성공: 기존 RefreshToken 폐기 (is_revoked=true)
          │                새 AccessToken + 새 RefreshToken 발급·저장
          │                원본 요청 재시도
          └─► 실패: logout() → /login 리다이렉트

[4] 로그아웃
  사용자 → POST /api/auth/logout
    └─► DB: RefreshToken 폐기 (is_revoked=true)
          Zustand 스토어 클리어 → /login 리다이렉트

토큰 스펙

항목 값
알고리즘 HMAC-SHA (jjwt 0.12.6)
Access Token 수명 30분
Refresh Token 수명 14일
Subject 클레임 userId
추가 클레임 email
프론트 저장 Zustand localStorage (chapchap-auth)
DB 저장 refresh_tokens 테이블 (is_revoked 플래그)

Refresh Token 로테이션

토큰 갱신 시 기존 Refresh Token을 즉시 폐기하고 새 토큰을 발급합니다 (로테이션). 탈취된 Refresh Token으로 재사용하는 공격을 방지합니다.

기존 RefreshToken → is_revoked = true (폐기)
새 AccessToken + 새 RefreshToken 발급
  └─► DB에 새 RefreshToken 저장
  └─► 프론트 Zustand 스토어 업데이트

SecurityConfig 필터 체인

JwtAuthenticationFilter가 UsernamePasswordAuthenticationFilter 앞에 등록되어 모든 요청을 먼저 처리합니다.

HTTP 요청 진입
  └─► JwtAuthenticationFilter (Spring Security 필터 체인)
        ├─► 공개 경로? → 통과 (SecurityContext 설정 없음)
        └─► 보호 경로?
              ├─► Authorization 헤더에서 Bearer 토큰 추출
              ├─► 서명 검증 + 만료 확인
              │     ├─► 유효 → SecurityContext 설정 → 컨트롤러 진행
              │     └─► 무효/만료 → 401 응답 (필터에서 직접)
              └─► 헤더 없음 → 401 응답
경로 접근 이유
/api/auth/** 공개 인증 전 접근 필요 (로그인·회원가입·토큰갱신)
/api/products/** 공개 로그인 없이 제품 탐색 허용
/api/ingredients/** 공개 로그인 없이 성분 조회 허용
/api/home/** 공개 날씨 팁 (공개 콘텐츠)
나머지 전체 JWT 필요 사용자별 개인 데이터 접근
미인증 요청에는 403이 아닌 401을 반환합니다. authenticationEntryPoint가 명시적으로 401을 응답하도록 설정되어 있습니다.

회원가입 접근 제어

회원가입(POST /api/auth/register)은 공개 경로이지만 두 가지 방어막이 적용됩니다.

1. 가입 코드 검증

요청 바디의 registrationToken이 서버 환경변수 REGISTRATION_SECRET과 일치해야 합니다. 환경변수가 비어 있으면 어떤 토큰을 입력해도 항상 거부됩니다.

조건 응답
IP 차단 상태429 Too Many Requests
토큰 불일치 (또는 REGISTRATION_SECRET 미설정)403 Forbidden
토큰 일치200 + AuthResponse (가입 완료)

2. IP 기반 브루트포스 차단

RegistrationAttemptService가 IP별 실패 횟수를 인메모리(ConcurrentHashMap)로 추적합니다.

항목 값
차단 임계값동일 IP에서 토큰 불일치 10회
차단 지속 시간1시간 (만료 후 자동 해제)
성공 시해당 IP의 시도 횟수 초기화
IP 추출 우선순위X-Forwarded-For 헤더 → RemoteAddr
저장소인메모리 (서버 재시작 시 초기화)
차단 기록은 인메모리이므로 서버 재시작 시 초기화됩니다. 프로덕션에서 장기 운용이 필요하다면 Redis 등 외부 저장소로 교체할 수 있습니다.
